# 发货管理 — 「智能拼车建议」功能设计方案

## 一、业务背景

当前发货管理中各发货单独立安排物流，实际运营中存在大量可拼车场景：
- 同一方向/途经路线的多个发货单 → 拼一车降低运输成本
- 单发货单体积/重量小 → 多单合并后刚好装满一辆车
- 发货日期相近、目的地相近 → 合并后统一安排

**目标**：系统自动分析待发货订单，给出拼车合并建议，由人工确认后一键合并。

## 三、数据模型变更

### 3.1 物料表新增字段（Material）

| 字段 | 类型 | 说明 |
|------|------|------|
| `approxWeightPerUnit` | Decimal(10,3) | 每基准单位大约重量(kg)，如1kg香菇≈0.001吨 |
| `approxVolumePerUnit` | Decimal(10,3) | 每基准单位大约体积(m³) |

> 这是估算值，不需要精确到每次称重。由产品管理员维护。

### 3.2 车辆表新增字段（Vehicle）

| 字段 | 类型 | 说明 |
|------|------|------|
| `maxLoadWeight` | Decimal(10,2) | 最大载重(吨)，如4.2米货车≈2吨 |
| `maxLoadVolume` | Decimal(10,2) | 最大容积(m³)，如4.2米货车≈16m³ |
| `length` | Decimal(5,2) | 车厢长度(米) |

> 由承运商管理维护。没有填写则不参与重量/体积约束。

### 3.3 客户地址表新增字段（CustomerAddress）

| 字段 | 类型 | 说明 |
|------|------|------|
| `lng` | Decimal(10,6) | 经度 |
| `lat` | Decimal(10,6) | 纬度 |

> 已有地址通过高德地理编码回填，新地址在地图选点时自动填入。

### 3.4 新增表：拼车建议（MergeSuggestion）

```sql
CREATE TABLE merge_suggestions (
  id              VARCHAR(36) PRIMARY KEY,
  suggestionNo    VARCHAR(30) UNIQUE,     -- 建议编号 MS20260706001
  sourceOrderIds  JSON,                   -- 源发货单ID列表 ["o1","o2","o3"]
  mergedOrderId   VARCHAR(36),            -- 合并后的发货单ID（确认后生成）
  totalWeight     DECIMAL(10,2),          -- 预估总重量(吨)
  totalVolume     DECIMAL(10,2),          -- 预估总体积(m³)
  totalKilometers DECIMAL(10,2),          -- 预估总里程
  estimatedCost   DECIMAL(12,2),          -- 预估运费
  routeDesc       TEXT,                   -- 路线描述 "淄博→济南→泰安→德州"
  matchScore      DECIMAL(5,2),           -- 匹配度 0-100
  matchReasons    JSON,                   -- 匹配原因 ["同一方向","发货日期相近"]
  status          VARCHAR(20) DEFAULT 'PENDING',  -- PENDING/CONFIRMED/CANCELLED
  confirmedById   VARCHAR(36),
  confirmedAt     DATETIME,
  createdBy       VARCHAR(20) DEFAULT 'SYSTEM',   -- SYSTEM/MANUAL
  createdAt       DATETIME DEFAULT NOW(),
  updatedAt       DATETIME
);
```

### 3.5 发货单表新增字段（ShippingOrder）

| 字段 | 类型 | 说明 |
|------|------|------|
| `mergedFromIds` | Json | 如果是合并单，记录源单号列表 |
| `isMerged` | Boolean (default false) | 是否为合并发货单 |

## 四、拼车匹配算法

### 4.1 算法流程

```
待发货订单（logisticsStatus != ARRANGED, shippingStatus != SHIPPED）
        │
        ▼
┌───────────────────────────────────────────────────┐
│  Step 1: 时间分组                                   │
│  按 shippingDate（±1天窗口）分组                    │
│  无 shippingDate → 取 SalesOrder.expectedDate       │
└───────────────────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────────────────┐
│  Step 2: 地理分组                                   │
│  用高德API计算 始发地→目的地 方向向量               │
│  同一方向象限的分一组（如都在东北方向）               │
│  同一路线途经的分一组（如都走G2京沪高速沿线）          │
└───────────────────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────────────────┐
│  Step 3: 路线优化排序                               │
│  对每组内的发货单按目的地距始发地远近排序             │
│  计算最优途经顺序 → 总里程最小                       │
│  （贪心：每次选距当前点最近的下一个目的地）            │
└───────────────────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────────────────┐
│  Step 4: 容量约束过滤                               │
│  累加重量/体积 ≤ 车辆最大载重/容积                   │
│  超出 → 拆分为多组                                  │
│  车辆信息缺失 → 跳过容量约束（仅按路线建议）           │
└───────────────────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────────────────┐
│  Step 5: 匹配度评分（加权）                          │
│  路线重叠度：40%                                    │
│  时间接近度：25%                                    │
│  容量利用率：20%（越高越好）                          │
│  同客户/同仓库加分：10%                              │
│  节省里程比例：5%                                   │
└───────────────────────────────────────────────────┘
        │
        ▼
    输出：TOP N 建议组（匹配度 ≥ 60% 的才显示）
```

### 4.2 匹配度评分细则

| 维度 | 权重 | 计算方式 |
|------|------|---------|
| **路线重叠度** | 40% | 合并后总里程 / 各自发货里程之和 的倒数，越接近1越好 |
| **时间接近度** | 25% | 1 - (最大日期差/7)，同一天=满分 |
| **容量利用率** | 20% | 总重量/车辆载重（无车辆数据默认满分） |
| **同客户/仓库** | 10% | 同一客户+5%，同一仓库+5% |
| **节省里程** | 5% | (各自总里程-合并里程)/各自总里程 |

评分范围 0-100，<60 分的不展示。

### 4.3 路线方向判断

用高德 API 方向规划实际验证：
1. 获取每单始发地→目的地方向向量（角度）
2. 任意两单方向夹角 < 45° → 可同组
3. 用方向规划 API 验证：始发地→所有目的地的最优路径是否合理
4. 如果路径需要大幅绕路（绕行比例 > 30%）→ 降分

### 5.2 合并发货交互

点击「合并发货」→ 弹出确认弹窗：

```

│  确认拼车合并                    [X]          │

│                                              │
│  ☑ SH-001  华绿生物  → 济南    1.5吨  ¥450   │
│  ☑ SH-003  华绿生物  → 泰安    1.2吨  ¥320   │

│  │ 总体积    18.5m³                     │   │
│  │ 总里程    520km                      │   │
│  │ 总运费    约¥1,200（原¥1,270，省¥70）  │   │
│  │ 路线      淄博→济南→泰安→德州         │   │
│  └──────────────────────────────────────┘   │
│                                              │
│  承运商：[华宇物流 ▼]   车辆：[鲁C·12345 ▼]  │
│  司机：[张三 ▼]         发货日期：[2026-07-06]│
│                                              │
│  [取消]  [确认合并]                           │
└──────────────────────────────────────────────┘
```

确认合并后：
1. 创建新的 ShippingOrder，isMerged=true
2. 将源单的明细复制到合并单的 shippingItems
3. 源单的 shippingStatus → MERGED（新增状态）
4. 合并单保持 logisticsStatus=PENDING，待后续安排物流
5. 保存 merge_suggestions 记录

合并发货单在列表中以特殊样式标记：
- 行前显示 🔗 图标

## 六、后端 API 设计

### 6.1 生成智能建议

```
POST /api/logistics/shipping-orders/merge-suggestions
```

请求参数：
```json
{
  "dateFrom": "2026-07-01",    // 筛选发货日期范围
  "dateTo": "2026-07-10",
  "providerId": "",            // 指定承运商（可选）
  "vehicleId": "",             // 指定车辆（可选，用于容量约束）
  "maxResults": 10
}
```

响应：
```json
{
  "success": true,

    "suggestions": [
```

### 6.2 确认合并

```
POST /api/logistics/shipping-orders/merge
```

请求：
```json
{
  "sourceOrderIds": ["o1", "o3", "o7"],
  "providerId": "p1",
  "vehicleId": "v1",
  "shippingDate": "2026-07-06",
  "suggestionId": "sug-001"
}
```

### 6.3 忽略建议

```
PUT /api/logistics/shipping-orders/merge-suggestions/:id/ignore
```

### 6.4 查看合并单详情

在现有 GET `/shipping-orders/:id` 响应中增加：
```json
{
  "isMerged": true,
  "mergedFromIds": ["o1", "o3", "o7"],
  "mergedFromOrders": [{...},{...},{...}]
```

## 七、实现计划

| 阶段 | 内容 | 预估复杂度 |
|------|------|-----------|
| **Phase 1** | 数据模型变更（Material/Vehicle/CustomerAddress 加字段、MergeSuggestion 表） | 低 |
| **Phase 2** | 后端拼车算法（时间/地理分组 + 路线优化 + 容量约束 + 评分） | **高** |
| **Phase 3** | 前端「智能建议」Tab + 建议列表 + 合并确认弹窗 | 中 |
| **Phase 4** | 合并单展示 + 历史建议管理 | 低 |

## 八、关键决策点（待确认）

| # | 决策项 | 选项 |
|---|--------|------|
| 1 | **重量/体积数据来源** | A. Material表加估算字段（简单） / B. 称重系统对接（精确但复杂） |
| 2 | **算法触发方式** | A. 手动点「生成建议」按钮（当前方案） / B. 定时任务自动生成 |
| 3 | **合并后源单状态** | A. 新增 MERGED 状态（不可再操作）/ B. 直接变更为已发货 |
| 4 | **合并单的库存锁定** | A. 合并时释放源单锁，重新锁合并单 / B. 保持源单锁不变 |
| 5 | **是否需要运输费用分摊** | A. 合并单统一运费，不拆分 / B. 按重量比例自动分摊到各源单 |
