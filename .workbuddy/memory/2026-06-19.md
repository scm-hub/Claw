## 全子系统 SSO Token URL 参数丢失修复 (09:52) ✅ 彻底解决
- **问题**：用户从 Portal 点击 AI 智能服务卡片后，子系统显示登录页而非自动登录。Console 显示 `hasToken: false, search: ""`
- **根因确认**：Vite 开发服务器在 SPA 模式下会**剥离 `location.search` 和 `window.location.search`**！即使 URL 栏显示 `?sso_token=xxx`，React 代码读到的 search 是空字符串。但完整 URL 仍可通过 `window.location.href` 获取
- **调试过程**：
  1. 加了不可关闭的顶部 DebugBar → 确认 `location.search=""` 和 `window.search=""`
  2. curl 测试后端 API 完全正常（token验证通过，返回用户数据）
  3. 用 Safari about:blank 清状态后测试 → href fallback 成功提取 token → 自动登录到仪表盘
- **修复内容**：
  1. **全部4个子系统的 SsoAutoLogin.jsx** 新增 `extractSsoToken()` 三级 fallback 函数：
     - Level 1: `react-router location.search`（标准方式）
     - Level 2: `window.location.search`（绕过 router）
     - Level 3: 从 `window.location.href` 手动解析（Vite SPA fallback）← 关键！
  2. **Portal Dashboard.jsx**：`window.open()` → 程序化 `<a>` 标签 click（更可靠，不被弹窗拦截器阻止）+ 防御性 null check
- **影响范围**：HRMS/SCM/MDM/AI 四个子系统全部更新（统一模式）
- **核心教训**：Vite dev server 的 SPA history fallback 会吃掉 query string，SSO 场景必须从 href 手动解析作为兜底

## 操作日志 IP 获取修复 (11:23) ✅
- **问题**：操作日志中 IP 全部显示 `::1`（IPv6 回环），用户认为 IP 没有获取到
- **根因**（三层缺配置）：
  1. Vite proxy 没有 `xfwd: true` → `x-forwarded-for` 头不被转发到后端
  2. Express 没有设置 `trust proxy` → `req.ip` 不会解析代理头
  3. `getClientIp` 没有 IPv6 归一化 → `::1` 和 `::ffff:1.2.3.4` 不好看
- **修复**：
  1. 4个 Vite 配置加 `xfwd: true`（portal/scm/hrms/mdm client）
  2. 4个 Express 服务器加 `app.set('trust proxy', true)`
  3. 5个文件的 `getClientIp` 改进：新增 `normalizeIp()`（`::1`→`127.0.0.1`，`::ffff:x`→`x`），增加 `x-real-ip` 头检查
  4. Portal `auth.js` 登录/访问系统日志也改用 `getClientIp` 替代 `req.ip`
- **验证**：通过 Vite 代理的请求 IP 从 `::1` 变为 `127.0.0.1`；子系统上报的自定义 IP 正确存储；局域网访问会显示真实 IP
- **改动文件**：8个配置文件 + 5个代码文件（operationLogger.js/auth.js/logs.js + 3个portalLogReporter.js）
