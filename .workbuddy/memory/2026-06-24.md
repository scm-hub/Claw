## 销售订单保存即锁定库存（占位）
- **问题**：新增销售订单保存后，可用库存没有减少。用户要求：可用库存的作用是占位，实际库存反映真实库存
- **原逻辑**：库存锁定在订单状态变为 CONFIRMED 时才执行
- **新逻辑**：保存订单（DRAFT）即锁定库存，出库（DELIVERED）时扣减物理库存+释放锁
- **改动文件**：`server/src/modules/sales/sales.routes.js`
- **改动内容**：
  1. POST /orders（创建）：保存前校验可用库存（聚合同物料数量）→ 创建订单(stockLocked=true) → 遍历 items 增加 lockedQty + 创建 StockLock 记录
  2. PUT /orders/:id（更新）：先释放旧锁(lockedQty-=, StockLock→RELEASED) → 删旧明细 → 校验新明细可用库存 → 创建新明细 → 重新锁定
  3. DELETE /orders/:id：删除前释放已锁库存
  4. PUT /orders/:id/status CONFIRMED：跳过重复锁定（保存时已锁），兼容旧数据(stockLocked=false)时仍执行锁定
  5. CANCELLED/DELIVERED：原有释放逻辑不变
- **后端已重启**

## 供应商评估自动取数算法修复
- **问题**：展开评估明细后 deliveryDetail/serviceDetail 为 null，qualityDetail 显示 0%
- **根因**：
  1. 准时交付率：算法过滤 `receiptStatus: ['RECEIVED','PARTIAL']`，但所有订单 receiptStatus 都是 PENDING（字段未及时更新），导致查不到任何订单
  2. 质量合格率：算法只认 `qcResult === 'PASS'`，但实际数据是 PENDING/INSPECTED，没有 PASS
  3. 服务响应：依赖同一个 orders 变量，订单查不到所以也算不出
- **修复**（`server/src/modules/extra/extra.routes.js`）：
  1. 准时交付率：去掉 receiptStatus 过滤，改为查所有订单后 filter 有 receiptDate 的（实际已交付的），取最早收货日期对比 expectedDate
  2. 质量合格率：PASS + INSPECTED 算合格，FAIL 算不合格，PENDING 跳过（未质检），detail 新增 pendingCount 字段
  3. 服务响应：改用 deliveredOrders 变量计算
- **前端**（`SupplierEvalList.jsx`）：质量明细展示从"检验总数"改为"已质检数"，新增 pendingCount 展示
- **已重新计算 2026-06 数据并更新数据库**
- **后端已重启 + 前端已构建**
