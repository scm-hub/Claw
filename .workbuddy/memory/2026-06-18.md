## 库存预占机制（解决下单后账物不符）
- **问题**：业务员下订单后库存应减少但仓库还没发货，导致其他业务员看到的库存不准确，可能超卖
- **根因**：数据库有 `lockedQty` 字段但后端未使用——CONFIRMED 只检查物理库存不锁定，DELIVERED 只扣物理不释放锁定，CANCELLED 不释放
- **修复后端** `sales.routes.js` 状态流转：
  - **CONFIRMED**：检查可用库存(qty-lockedQty) >= 订单量 → lockedQty += 订单量（锁定）
  - **DELIVERED**：qty -= 订单量（扣物理库存）+ lockedQty -= 订单量（释放锁定）+ 创建出库记录+应收
  - **CANCELLED**：如果 stockLocked=true → lockedQty -= 订单量（释放锁定）+ stockLocked=false
- **修复后端** `wms.routes.js`：inventory API 返回新增 `availableQty = qty - lockedQty`
- **前端更新**：
  - PC 销售订单明细表：从「当前库存」改为「物理库存」+「可用库存」两列，可用不足时⚠
  - 移动端销售订单：物料下方显示 物理:XX / 可用:XX / (锁定YY) / ⚠可用不足
  - 移动端库存页：显示 物理/可用/锁定 三个数字
  - PC 新增「库存台账」页面(`wms/InventoryList.jsx`)：物理/锁定/可用三列 + 占用率进度条
- **E2E 测试通过**：
  - 创建前: 物理120 锁定0 可用120
  - 确认后: 物理120 锁定10 可用110 ✅
  - 取消后: 物理120 锁定0 可用120 ✅

### Prisma Schema 修复（51→0 错误）
- **问题**：45 个模型的 Prisma Schema 有 51 个验证错误
- **主要错误类型**：
  1. 一对一关系缺少 `@unique`（User.employeeId, PurchaseReceipt.batchId, AccountsReceivable.invoiceId 等）
  2. 缺少反向关联字段（Employee 缺少 Warehouse/PriceList/PurchasePlan/TemperatureAlert 等反向关系）

  4. 可选标量字段配非可选关系（Sensor.warehouse, BarcodeRecord.material）→ 关系改为可选
  5. Department.manager 缺少 fields/references 参数
  6. PurchaseReceipt.material 使用空 `@relation(fields: [], references: [])` → 添加 materialId 字段
  7. Contract 多态关系使用同一 partyId 字段 → 拆分为 supplierId + logisticsId 独立 FK
  8. AlertRecord 与 User 有多个关系但 targetUser 未命名 → 添加 "AlertTargetUser" 名称
  9. Material.batchTrackings 无对应 BatchTracking.materialId → 移除该无效反向关联
  10. Supplier.purchaseReceipts 无对应 PurchaseReceipt.supplierId → 移除该无效反向关联
