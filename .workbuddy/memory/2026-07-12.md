## HRMS→金蝶 部门推送功能修复

### 原始问题
- 推送部门到金蝶后：① 财务部/人力资源部上级部门不显示 ② 数据状态为草稿(A)非已审核(C)

### 根因分析
1. **父部门映射时机**：`pushDepartments` 循环内推送根部门后，`deptFNumberMap` 要到函数返回才更新，子部门推送时找不到父部门的金蝶编码 → FParentID 为空
2. **审核状态**：`save('BD_Department', data, true)` 的 `isAutoSubmitAndAudit` 对 BD_Department 表单不生效，部门仍为 Status A

### 修复方案
- **父部门**：将 `deptFNumberMap[dept.hrmsId] = fNumber` 从函数末尾移到循环内每个部门 Save 成功后立即执行
- **审核**：Save → 收集 FID → 批量 Submit + Audit（Ids 用逗号分隔字符串如 `"3191401,3191402"`）
- **关键发现**：Kingdee Submit/Audit API 的 Ids 参数必须为逗号分隔字符串，不能用数组

### 最终结果
5 个部门全部推送成功，Status=C（已审核），子部门正确显示上级 QH_DEPT_001

## 产品管理编码保存问题修复
### 问题1：新增产品时编码被后端覆盖
- 根因：`POST /materials` 中 `data.code = await genCode(...)` 无条件覆盖前端传入的金蝶编码
- 修复：`if (!data.code) data.code = ...`

### 问题2：编辑时编码不生效
- 根因：`PUT /materials/:id` 将 `code` 从请求体中解构排除，不会写入数据库
- 修复：去掉 `code` 的排除

### 问题3：编辑保存时编码唯一冲突（materials_code_key）
- 根因：编辑时修改编码（如选择另一个金蝶物料），但新编码已被其他物料占用，直接触发 Prisma P2002 原始错误
- 修复：在 `prisma.material.update` 前增加 `findFirst` 预检查，若新编码被其他物料使用则返回 400 中文提示

### 问题4：新增产品时编码唯一冲突（materials_code_key）
- 根因：产品 POST 只给编辑加了唯一性预检查，新增没有
- 修复：在 `POST /materials` 中，若前端传入的编码已存在，则返回 400 中文提示
