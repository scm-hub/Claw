## 成本价计算单位折算优化（12:00）

### 需求
费用登记中成本价计算公式优化：把参与计算的每条数据单位都折算成基准单位，然后再执行计算

### 问题分析
各来源 StockMovement.qty 的存储单位不一致：
- PURCHASE_RECEIPT（入库）：已在写入时通过 `purchaseQtyToBase` 转为基准单位 ✅
- SALES_ORDER（销售出库）：已在写入时通过 `salesQtyToBase` 转为基准单位 ✅
- SHIPPING_ORDER（发货出库）：**未转换**，存储的是销售单位数量 ❌
- MANUAL（手动出入库）：基准单位（假设）✅
- StockTakeItem.diffQty（盘点损耗）：来源于库存数量差，基准单位 ✅

另外发现 `initialPurchasePrice`（采购期初价）是采购单位单价，但在用作 `beginningPrice` 和入库 `unitPrice` fallback 时未转换为基准单位单价

### 修改文件
- `server/src/modules/cost-price.scheduler.js`
  1. import 加入 `salesQtyToBase`
  2. material select 加入 `salesUnit`、`salesConversionFactor`
  3. 出库查询 select 加入 `refType` 字段
  4. 出库数量计算：SHIPPING_ORDER 类型用 `salesQtyToBase(rawQty, mat)` 转换为基准单位
  5. `initialPurchasePrice` 用作 `beginningPrice` fallback 时用 `purchaseUnitPriceToBase` 转换
  6. `initialPurchasePrice` 用作入库 `unitPrice` fallback 时同样转换
  7. 添加注释标注各数据来源的单位

### 验证
- scm-server 重启成功，成本价计算自动执行：17个分组全部成功
- 费用登记页面的"重新计算"按钮调用同一函数，路径全覆盖

## 审批流架构升级：节点+连线模式（00:12）
### 需求
- 当前逻辑：分支匹配后按数组顺序走完所有后续节点，不符合"分支命中后直接结束"的预期
- 目标：像泛微OA一样，每个节点通过连线决定下一节点，hit分支后直接走对应节点→结束

### 改动
1. **workflow.service.js 引擎**:
   - `normalizeTemplate()`: 兼容新旧格式
   - `findConnection()` / `resolveByConnections()`: 连线路由
   - `startFlow`: 新格式走连线路由，旧格式保留 branches 逻辑
   - `approveTask`: 新格式用 `resolveByConnections` 找下一节点，目标为 `end` 则完结
   - 旧格式完全兼容，引擎自动检测
2. **FlowTemplates.jsx 前端**:
   - 新增"连线配置"面板：起点节点 → 目标节点 → 条件（字段下拉+运算符+值）
   - 支持 `end` 目标（流程完成）
   - 节点字段从 `nodeId/nodeName` 迁移为 `id/name`
   - `handleSave` 打包为 `{ nodes, connections }` 格式
3. **种子数据更新**为连线格式
4. **模板 DB 已更新**为连线格式

### 新模板流程
```
1(条件) → marginRate>=25 → 2(部门经理) → end
        → marginRate>=20 → 3(上层领导) → end
        → default        → 4(总经理)   → end
```
每个分支命中后审批完对应节点就结束，不再走后续节点。
