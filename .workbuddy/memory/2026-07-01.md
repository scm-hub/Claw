## 修复采购计划智能建议逻辑问题
- **问题**: 用户反馈智能建议只出一条记录。根因分析发现：
  - 数据库无销售出库记录时，`dailyAvgSales=0`，`safetyStock=0`，但 `suggestedQty=Math.max(0,0-(-110))=110` 仍>0 被推送，导致香菇（库存-110）出建议
  - 其余物料库存=0、salesQty=0时 safetyStock=0、suggestedQty=0，条件 `suggestedQty>0 || availableStock<safetyStock` 两个都不满足，不出建议
  - 前端 `handleLoadSuggest` 中 `res.data.suggestions` 是正确的 api.js 一层嵌套访问模式，不需要改
- **修复**:
  - 后端算法调整：无销售历史时日均按1件估算(`effectiveDailyAvg`)，避免 safetyStock=0 无意义
  - 建议采购量公式改为 `suggestedQty = maxStock - availableStock`（补到最大库存水位，而非安全库存）
  - 出建议条件简化为 `suggestedQty > 0`（库存不足才建议采购）
  - 前端算法说明文案同步更新
- 构建前端 + 重启 scm-server 与 scm-pc-preview 完成

## 采购计划数据隔离逻辑修改：非采购角色+采购员绑定可见
- **问题**: 吕永权(role=FINANCE_MANAGER)被 PurchaserAssignment 绑定为采购员，但看不到分配给自己的采购子计划
- **根因**: 数据隔离方案三对非采购角色直接返回 `{ id: '___NO_ACCESS___' }`，不看业务关联
- **修改**:
  - `getPlanDataFilter()` 和 `hasPlanViewAccess()` 改为 async（需查 PurchaserAssignment 表）
  - 新增 `isPurchaserAssignee(userId)` 函数（5分钟缓存TTL），查 PurchaserAssignment 表判断是否有采购员绑定
  - 非采购角色但有采购员绑定 → 返回 `{ assigneeId: userId }`，可见分配给自己的子计划
  - 非采购角色无采购员绑定 → 返回 `{ id: '___NO_ACCESS___' }`，返回空数据
  - 详情路由的 else 分支改为检查 assigneeId=自己（不再直接403）
  - 提交审批路由增加 authorize 中间件（SUPER_ADMIN/PURCHASE_MANAGER/PURCHASE_STAFF），非采购角色即使能看到计划也不能提交审批
  - 所有调用点 `getPlanDataFilter(req.user)` → `await getPlanDataFilter(req.user)`，`hasPlanViewAccess(req.user)` → `await hasPlanViewAccess(req.user)`
- **测试验证**:
  - 吕永权(FINANCE_MANAGER+采购员绑定) → 看到2条assigneeId=自己的子计划 ✅
  - 吕永权子计划列表 → 看到2组allocated ✅
  - 吕永权智能建议 → totalItems=11 ✅
  - 吕永权详情 → 能看分配给自己的子计划 ✅
