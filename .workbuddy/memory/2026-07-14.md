## 产品管理等级返填
- 需求：SCM 新增产品时，选中金蝶物料后把等级反填到等级字段
- 实现链路：MDM 同步物料时携带 grades → SCM 前端匹配 → gradeIds
- **问题与修复**：
  1. `pullMaterialsFromKingdee` 用 `adapter.pullMaterials()` 同步，`extra` 不含 grades → 改用 `pullMaterialsWithGrades()`
  2. `pullMaterialsWithGrades` 原版用 `FIDAuxProp` 字段（金蝶报不存在）→ 改用 `FAuxPropertyId`+`FAuxPropertyId.FName`
  3. 原版字段名（`.FNUMBER`）和 `pullMaterials` 不一致导致 materialGroup 为空 → 对齐字段名
  4. 等级查询用 `FID.FName` 查不到 → 改用 `FID.FNUMBER='DJ'`（金蝶等级组编码）
  5. 测试库 material_grades 表为空 → 从正式库同步 60 条
- **后续 bug**：所有物料都拿到 DJ 组的 60 个等级全量 → 蒲坤反馈与金蝶实际显示不一致
  - 金蝶 WebAPI 没有 FIDAuxProp 字段，无法按物料限定具体等级
  - 最终方案：每条物料 grades 置空，用户手动选等级
- 涉及文件：Claw-test/mdm/server/src/services/{kingdee-adapter.js,kingdee-sync.js}；xdj_scm_db_test.material_grades
- **金蝶等级 group 编码 DJ**：物料等级（物料分类辅助属性组）的 FNumber 固定为 DJ
- **金蝶辅助属性字段**：BD_MATERIAL 上的 FAuxPropertyId 是该物料关联的辅助属性值 ID（数字），FAuxPropertyId.FName 是该辅助属性值的显示名称（"物料等级"）；按物料限定的具体等级原应在 FIDAuxProp 字段（FID+FValue 结构），但 WebAPI 实际不可用

## 采购员分配架构重构：User → Employee（按用户反馈）
- **用户反馈**："为什么你要用 SCM 的 User 进行验证？采购员跟这个用户是否登陆是没有关系的"
- **根因**：原 PurchaserAssignment.userId 是 User 表外键，导致"未登录过 SCM 的人不能被分配"的不合理业务约束
- **架构修复**：PurchaserAssignment 外键从 User 改为 Employee
  - **Schema**：`userId String @unique` → `employeeId String @unique`，relation User → Employee
  - **后端路由**：
    - `POST /purchaser-assignments` 接受 `employeeId` 替代 `userId`
    - `GET /purchaser-assignments` 列表/详情 include 改为 `employee`
    - `GET /purchaser-assignments/my-materials` 通过当前 User.id 反查 employeeId
    - `isPurchaserAssignee(userId)` 缓存结构改为 userId Set（内部分两步：assignment→employeeId→user）
    - 删除引用验证的 PurchasePlan 查询改用 User.employeeId 映射
  - **`/purchaser-users` 简化**：不再需要返回 userId，移除 _noScmUser 标记
  - **purchase-plan.scheduler.js**：allocatePlan 时反查 User（assigneeId 仍需 User.id，因 PurchasePlan 体系用 User）
  - **前端**：
    - 去掉 `_noScmUser` 校验/禁用/标识
    - PurchaserAssignment 列表/详情改用 `row.employee`
    - PurchasePlanList 转发弹窗数据源改为 `a.employee`，显示改为 `{name} ({empNo}) - {department}`
- **数据库**：prisma db push --accept-data-loss（测试表为空，零数据损失）
- **测试验证**：王小明（已登录SCM）和吕永权（未登录SCM）都能成功创建采购员分配
