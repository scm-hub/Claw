## HRMS→金蝶 部门/员工推送功能

### 问题诊断
- 原有 `pushDepartments`/`pushEmployees` 代码存在严重 bug：
  - 部门 FNumber 用了 MDM UUID（金蝶不接受超长编码）
  - 员工用了 `FNumber` 字段（金蝶 BD_Empinfo 主键是 `FStaffNumber`）
  - 组织 ID 硬编码为 `'100'`（实际应为 `'10001'` = 山东七河生物科技股份有限公司）
  - 部门父级引用和员工部门引用使用了 MDM UUID 而非金蝶编码
  - DataMapping 唯一约束名写错（`entityType_systemCode_masterId` → `entityType_masterId_systemCode`）

### 修复内容（3 个文件）
| 文件 | 改动 |
|------|------|
| `kingdee-adapter.js` | `pushDepartments` 改用 `QH_DEPT_xxx` 编码、组织 `10001`、支持 deptFNumberMap 查父级；`pushEmployees` 改用 `FStaffNumber`、`FDEPARTMENT`、组织 `10001` |
| `kingdee-sync.js` | `pushDepartmentsToKingdee` 增加名称匹配金蝶已有部门、存在则跳过；`pushEmployeesToKingdee` 增加 FStaffNumber 匹配、存在则跳过；两函数均自动保存 DataMapping |
| `kingdee.js` | `push-all` 路由传递 `deptFNumberMap` |

### 名称匹配误判问题
- 问题：商超事业部和餐饮事业部在金蝶中名称匹配到了预存的 `00000001715`/`00000001716`，被「存在则跳过」拦截
- 根因：金蝶中同一部门名称在不同组织下重复存在，名称匹配不可靠
- 修复：去掉预查询名称匹配，只信任 DataMapping（我们自己推送过的）；若金蝶报名称冲突，自动查找已有部门并建立映射
- 结果：餐饮事业部新建成功（QH_DEPT_002），商超事业部因金蝶 org 100066 中已有同名部门，映射到 00000001715（用户需要确认是否可用这个已有部门）

### 测试结果
- 部门：5/5 ✅（金蝶中已存在，名称匹配成功，DataMapping 已保存）
- 员工：11/11 ✅（金蝶中已存在，FStaffNumber 匹配成功，DataMapping 已保存）
- DataMapping：5 部门 + 11 员工 = 16 条

### 仪表盘使用
- MDM 仪表盘 → 金蝶 Tab →「推送部门到金蝶」/「推送员工到金蝶」
- 或「全部推送」（部门+员工一起）
