## SSO 跳转修复 — 子系统 API 路径适配网关

### 问题
从 Portal 点击子系统卡片后，SCM/HRMS/MDM 都跳到各自登录页，SSO 自动登录失败。

### 根因
各子系统前端 API baseURL 硬编码为 `/api`，通过网关访问时：
- 前端发 `POST /api/auth/sso-login` → 网关 `/api` 路由到 **Portal 后端(4001)**
- Portal 后端没有 `/auth/sso-login` 路由 → SSO 失败 → 跳登录页

### 修复
将三个子系统+移动端的 API baseURL 改为动态拼接：
```js
// 修改前
const BASE_URL = '/api';
// 修改后
const BASE_URL = import.meta.env.BASE_URL.replace(/\/$/, '') + '/api';
```
- dev 模式：BASE_URL=`/` → API base=`/api`（Vite proxy 正常）
- prod 网关：BASE_URL=`/scm/` → API base=`/scm/api`（网关代理到 4003）

同时修复 401 重定向路径：`'/login'` → `import.meta.env.BASE_URL + 'login'`

| `xdj-scm/client/src/lib/api.js` | BASE_URL + 401 redirect |

## 外网跳转超时修复

### 问题
外网通过 Portal 点击子系统卡片时"跳转不过去，超时"。

### 根因
`Dashboard.jsx` 的 `handleSystemClick` 是 `async` 函数，先 `await api.post('/auth/access-system', ...)` 再打开新标签页：
1. 如果 API 调用慢（外网延迟），跳转被阻塞
2. 如果返回 401（令牌过期），axios 拦截器执行 `window.location.href = '/login'`，当前页跳走，新标签页打不开
3. axios 超时 15s，用户感觉就是"超时"

### 修复（两层）
1. **Dashboard.jsx**：改为同步函数，先打开新标签页，再异步发 access-system 日志（`.catch(()=>{})`）
2. **api/index.js**：401 拦截器加 `_skipAuthRedirect` 开关，access-system 调用传此标记跳过登录重定向

### 端到端验证
- Portal 登录 → 获取 SSO token（563字符 JWT）✅
- SCM/HRMS/MDM 页面加载 200 ✅（< 1ms）
- SCM/HRMS/MDM SSO API 返回 success + local token ✅

## 推送采购计划报错深度排查（持续中）

### 问题
需求汇总"推送生成采购计划"点击确认后，弹窗始终显示「推送失败 / 未知错误，请重试」，即使后端 curl 测试完全正常。

### 排查过程（3轮）
1. **第1轮**：前端原生 fetch 的 error 对象无 `.response` 属性 → 改进 catch 块显示 `e.message` + console.error 日志 → 用户反馈**还是空白错误**
2. **第2轮**：加 `[errType]` 前缀 + 更详细的 console.error 诊断信息（selectedCount/validItemsCount/requestBodySize）→ 用户反馈**仍然"未知错误"**
3. **第3轮（当前）**：
   - **api.js 底层增强**：`fetch()` 加 try-catch 捕获网络层错误；`res.json()` 改为先读 text 再 parse，非 JSON 时截取前 200 字符显示
   - **gateway.js 加 no-cache 头**：Express static 文件服务对所有 .js/.css/.html 加 `Cache-Control: no-store, no-cache, must-revalidate, proxy-revalidate`
   - **重启网关**：确保新构建的 JS（index-BQFD-fdS.js）被浏览器加载
   - curl 验证 no-cache 头生效 ✅

### 关键发现
- 后端 API 完全正常（curl 用真实 materialId 测试成功创建 PP202606219BJ3DP）
- 问题几乎确定是**浏览器缓存了旧的 JS 文件**（旧代码的错误处理不完善，导致 message 为空）
- 网关已加 no-cache 头并重启，等待用户强刷验证

### 🔴 最终根因（第4轮发现）
**不是缓存问题、不是网络问题、不是后端问题——是前端代码逻辑错误！**

```js
// ❌ 错误代码
const res = await api.post('/sales/plans/push-to-purchase', {...});
setPushResult({ ...res.data, ... });
// res = { success: true, data: { planNo, items }, message: "创建成功" }
// ...res.data 展开了内层 data，但丢失了 success 字段
// → pushResult.success = undefined（falsy）→ 走了错误分支！→ 显示"未知错误"

// ✅ 修复
setPushResult({
    success: res.success,   // true
    data: res.data,         // { planNo, items }
    message: res.message,   // "创建成功"
});
```

**教训**：api.js 的 `request()` 函数返回的是整个响应体 `{ success, data, message }`，不是 axios 那种 `{ data: { success, data, message } }` 的结构。前端代码误以为 `res.data` 就是全部数据，直接展开导致 `success` 字段丢失。

### 排查反思
1. 前三轮一直在优化**错误捕获和显示**（catch 分支），但实际错误发生在 **then 分支**（成功回调），catch 根本没执行
2. 应该第一时间检查 `pushResult` 的实际值——如果走的是错误分支但 catch 没触发，说明是**成功分支的数据处理有问题**
3. **排查优先级应该是**：先确认浏览器加载了新代码（版本标记）→ 确认 API 请求实际发出且成功 → 检查成功回调的数据处理逻辑 → 最后才优化错误处理
