### MDM 全量同步异步化（解决 30s 超时）
- 问题：点击「全量拉取物料产品」15,000+ 条数据，30s 超时
- 根因：前端 axios 默认 30s 超时、网关 3 分钟超时，均不足以完成全量同步
- 改造：
  - 后端新建 `sync-task-manager.js`：以 `sync_log` 表为持久化存储，内存仅作 RUNNING 等临时状态补充；支持 createTask/startTask/updateTask/finishTask/failTask/getTask/getTasks
  - kingdee.js 新增：`POST /sync-tasks` 启动异步任务，`GET /sync-tasks/:taskId` 轮询状态，`GET /sync-tasks` 列表
  - 异步任务：创建 sync_log(PENDING) → 后台运行 → 更新 sync_log(SUCCESS/PARTIAL/FAILED)
  - 前端 Dashboard.jsx：全量按钮调用 `/kingdee/sync-tasks`，启动后显示「任务进行中」Alert；useEffect 每 3 秒轮询状态，完成后刷新仪表盘并提示结果
- 结果：全量同步不再受前端 30s 限制，任务在后台完成

### MDM 全量同步两 Bug 修复
- **Bug1**: `updateTask is not defined` — kingdee.js 中 runAsyncPullTask() 使用了 updateTask 但未从 sync-task-manager.js 导入
  - 影响：异步任务拉取数据后崩溃，sync_log 标记为 FAILED（但数据 Upsert 实际已完成）
- **Bug2**: extra 字段（baseUnit/purchaseUnit/salesUnit/materialGroup/auxProperty）全为空字符串
  - 根因：金蝶 ExecuteBillQuery 返回引用字段值是**数字**（如 `10101`、`114913`），不是预期中的 `{FNumber: 'xxx'}` 对象
  - parseRef 只处理了 string 和 object 类型，遇到 number 返回空字符串
  - 修复：parseRef 增加 `typeof val === 'number'` 分支，`return String(val)`
- 修复后需用户重新点击「全量拉取物料产品」按钮触发同步（约 478,000 条金蝶物料，需 7-8 分钟）

### MDM 异步任务轮询状态不更新修复
- 问题：同步完成（DB 状态 SUCCESS）后，仪表盘仍显示「全量同步任务进行中：物料，状态：等待中」
- 根因：
  - `sync-task-manager.js` 的 `getTask` 使用 `memory.status || log.status`，当内存状态为 PENDING 时会覆盖数据库的 SUCCESS
  - `startTask/updateTask` 只更新内存，没有同步数据库，PM2 多实例时状态不一致
  - mdm-server 实际运行了 2 个 PM2 实例，内存 `memoryTasks` 不共享
- 修复：
  - `sync-task-manager.js` 重写状态优先级逻辑：`getTask/getTasks` 取 memory 和 DB 中更高级的状态（PENDING < RUNNING < PARTIAL < SUCCESS/FAILED）
  - `startTask/updateTask` 同时把数据库状态更新为 RUNNING
  - `ecosystem.config.js` 中 mdm-server 显式设置 `instances: 1` 并通过 `pm2 scale mdm-server 1` 缩容
  - 验证 `/api/kingdee/sync-tasks/:taskId` 返回 `status: SUCCESS, progress: 100, total: 478659, processed: 478659`
- 建议：仪表盘提示不会自动消失，需要用户按 `Cmd+Shift+R` 刷新页面，或等待最多 3 秒轮询自动更新
