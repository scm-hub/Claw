## 入库管理质检附件上传 - 全链路修复
- **根因1**：`client/src/lib/api.js` 的 `api.post()` 对 FormData 执行 `JSON.stringify(body)` → 变成空 JSON `"{}"`
  - 修复：FormData 检测 + 原样传递 + 不手动设 Content-Type
- **根因2**：用户走网关 5174 端口，读 client/dist/ 构建产物，源码修改后未 build 导致修复不生效
  - 修复：每次改源码后必须 `npm run build`
- **根因3（本次）**：gateway.js 的 `/uploads` 代理用 http-proxy-middleware 会**自动剥离匹配路径** `/uploads`
  - 浏览器请求 `/scm/uploads/qc/xxx.jpg` → 网关转发到后端变成 `/qc/xxx.jpg`（缺少 /uploads 前缀）
  - 后端 `express.static('uploads')` 挂载在 `/uploads` 路径下 → 找不到路由
  - **修复**：gateway.js 加 `pathRewrite: (p) => '/uploads' + p`；vite.config.js 加 `rewrite: (p) => '/uploads' + p`
- **附件查看链接跳转 Portal**：ReceiptList 新增 `getAttachmentUrl()` 函数拼接 BASE_URL + 附件路径
- **附件代理路径剥离**：gateway.js / vite.config.js 的 /uploads 代理加 pathRewrite/rewrite 拼回前缀

## 入库单展开明细不生效 - 网关缓存修复
- **问题**：用户在入库管理页面点击展开行无反应，无法看到入库单明细
- **排查**：数据库正常（3条明细）、API 正常返回 items、构建产物包含展开行代码、构建时间晚于源码修改时间
- **根因**：gateway.js 的 SPA fallback（`spaFallback`）响应未设置 no-cache 头 — 原来的 no-cache 中间件用正则 `/\.(js|css|html|...)/` 匹配请求路径，但 SPA 路由如 `/scm/purchase/receipts` 没有文件扩展名，不匹配正则，导致 HTML 响应无缓存头 → 浏览器缓存旧 HTML → 引用旧 JS 文件（无展开行功能）
- **修复**：gateway.js 改为对所有响应无条件设置 `Cache-Control: no-store, no-cache, must-revalidate, proxy-revalidate`（去掉正则匹配条件），Portal 部分同样修复
- **重启**：网关 5174 + SCM 后端 4003 均已重启
- **用户操作**：需要浏览器硬刷新（Cmd+Shift+R）清除旧缓存
